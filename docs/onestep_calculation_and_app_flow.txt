ONESTEP 계산구조 및 앱 작동방식 정리

작성 기준
- 분석 대상: 현재 ONESTEP 프로젝트 코드베이스
- 목적: ChatGPT에 업로드하여 발표자료/PPT 제작에 활용할 수 있는 서비스 흐름 중심 설명
- 원칙: 실제 코드에서 확인되는 내용만 작성
- 확인되지 않는 기능은 “현재 코드에서 확인 불가”로 표시


1. 서비스 개요

서비스 목적:
ONESTEP은 OD 교통수요 데이터를 바탕으로 교통 흐름이 큰 지역 쌍을 찾고, 그 사이에 신규 도로 또는 터널 후보 노선을 자동으로 계산해 비교하는 서비스이다.

이 앱이 해결하려는 문제:
- 도로 또는 터널 사업 후보지를 사람이 일일이 찾기 전에, 교통수요가 큰 연결 구간을 먼저 추려낸다.
- 기존 도로망을 이용했을 때의 경로와 DEM 지형 기반 신규 경로를 비교한다.
- 후보 노선별 거리, 시간 절감, 예상 편익, 공사비, 토지보상비, 경제성 점수를 한 화면에서 확인할 수 있게 한다.

사용자 입력:
- 기본 OD CSV 또는 사용자가 업로드한 OD CSV
- 선택 시나리오 OD 데이터
- 분석 지역 필터
- OD 후보 연결쌍 개수
- 상위 수요 필터
- 저영향 후보 정점 제거 비율
- 샘플 크기
- 후보 노선 산정 개수

사용자가 받는 결과:
- OD 기반 후보 정점 목록
- 후보 정점 간 연결쌍 목록
- 기존 도로 기준 경로
- DEM 기반 신규 직접 노선
- 기존 도로 일부와 신규 우회 구간을 섞은 혼합 노선
- 후보 노선별 구간 분류: 기존 도로, 접속 도로, 신규 지상도로, 터널
- 후보 노선별 비용, 편익, B/C, 순위, 지도 표시용 geometry

신규 도로/터널 후보 노선을 산정하는 전체 목적:
교통수요가 큰 지역 간 연결을 대상으로, 기존 도로만 이용하는 기준 경로보다 거리 또는 시간이 의미 있게 개선되는 신규 도로·터널 후보를 찾고, 비용과 편익을 함께 비교해 초기 사업 검토용 우선순위를 만드는 것이다.


2. 앱 작동 흐름

웹 화면 기준 전체 흐름:
1) 사용자가 OD 데이터 조건을 설정한다.
- 기본 OD 사용 여부, 업로드 CSV, 시나리오 OD, 지역 필터, 후보 연결쌍 개수 등을 선택한다.
- 관련 프론트엔드 파일: frontend/src/App.jsx

2) 프론트엔드가 OD 후보 생성 API를 호출한다.
- POST /api/od-candidates
- FormData 형식으로 top_node_limit, edge_limit, include_base_od, use_region_filter, selected_regions 등을 전송한다.
- 관련 함수: runCandidateGeneration

3) 백엔드가 OD 데이터를 읽고 후보 정점을 만든다.
- OD CSV에서 좌표 또는 행정동 코드를 읽는다.
- 좌표가 없으면 data/processed/admin_dong_coordinates.csv의 행정동 좌표 lookup을 사용한다.
- 출발지와 도착지를 모두 가중치가 있는 점으로 변환한다.
- K-means 클러스터링으로 수요 중심 후보 정점을 만든다.
- 가까운 후보 정점은 3km 기준으로 병합한다.
- 관련 파일: backend/app/services/od_candidate_generation.py

4) 백엔드가 후보 연결쌍을 만든다.
- 각 OD의 출발지와 도착지를 가장 가까운 후보 정점에 매칭한다.
- 후보 정점 쌍별로 OD 유량을 합산한다.
- 직선거리 5km 미만 또는 100km 초과 쌍은 제외한다.
- estimated_flow가 큰 순서대로 후보 연결쌍을 정렬한다.

5) 프론트엔드가 후보 정점과 연결쌍을 지도와 결과 패널에 표시한다.
- RouteMap 컴포넌트가 카카오 지도 위에 후보 정점, 연결쌍, 이후 생성된 노선 segment를 표시한다.
- 관련 파일: frontend/src/App.jsx

6) 사용자가 “도로·터널 후보 생성” 기능을 실행한다.
- 프론트엔드가 POST /api/candidate-routes를 호출한다.
- OD 후보 생성 결과의 nodes, edges, route_limit, 지역 필터 정보를 JSON으로 보낸다.
- 관련 함수: runCandidateRouteAnalysis

7) 백엔드가 후보 연결쌍별 노선 계산을 시작한다.
- estimated_flow가 큰 edge부터 처리한다.
- 각 edge마다 기존 도로 기준 경로, 신규 직접 경로, 혼합 우회 경로를 생성한다.
- 관련 파일: backend/app/services/candidate_route_pipeline.py

8) 기존 도로망 A* 기준 경로를 계산한다.
- 출발 후보 정점과 도착 후보 정점에서 가장 가까운 도로 노드를 찾는다.
- road_nodes, road_links를 이용해 도로망 그래프를 만든다.
- 도로망에서 A* 탐색으로 기존 도로 경로를 계산한다.
- 관련 파일: backend/app/services/road_graph_routing.py, backend/app/services/road_network.py

9) DEM 기반 신규 노선 경로를 계산한다.
- 출발·도착 좌표 주변에 비용 격자를 만든다.
- DEM 고도에서 경사도를 계산하고, 경사도·하천·도시/건물·기존도로 접근성에 따라 격자 비용을 조정한다.
- 비용 격자에서 A*로 최저비용 경로를 찾는다.
- 관련 파일: backend/app/services/cost_grid.py, backend/app/services/pathfinding.py

10) 경로 segment를 분류한다.
- 각 경로를 기존 도로, 신규 지상도로, 터널 등으로 분류한다.
- 도로망 기준 경로의 양 끝에는 후보 정점에서 도로망까지의 connector가 붙는다.
- 하천 교량 segment는 현재 별도 분류되지 않는다.
- 관련 파일: backend/app/services/route_segments.py

11) 비용을 계산한다.
- 신규 지상도로 길이, connector 길이, 터널 길이, 토지보상비를 바탕으로 비용을 계산한다.
- 기존 도로 구간은 건설비에 포함되지 않는다.
- 관련 파일: backend/app/services/cost_model.py, backend/app/services/land_compensation.py

12) 편익과 점수를 계산한다.
- 기존 도로 기준 경로 대비 후보 노선의 거리 절감과 시간 절감을 계산한다.
- 절감률로 전환율을 계산하고, estimated_flow 중 전환된 통행량만 편익에 반영한다.
- B/C, 순편익, 후보 점수를 계산한다.
- 관련 파일: backend/app/services/cost_model.py, backend/app/services/route_economics.py

13) 후보 노선을 순위화한다.
- 기존 도로 기준 경로는 비교 기준으로 유지되지만 ranked_routes에는 신규/혼합 후보만 들어간다.
- candidate_score와 economic_score를 이용해 정렬한다.
- 관련 파일: backend/app/services/route_economics.py

14) 프론트엔드가 지도와 결과 패널에 표시한다.
- 지도에는 segment_type별 색상으로 기존도로, 신규도로, 접속도로, 터널을 표시한다.
- 결과 패널에는 순위, 길이, 수요, 거리 절감, 비용, 경제성 점수를 표시한다.
- 관련 파일: frontend/src/App.jsx


3. 주요 계산구조

OD 데이터 처리 방식:
- 사용 파일: backend/app/services/od_candidate_generation.py
- 관련 함수: load_od_data, _detect_flow_columns, _row_to_record, _read_candidate_records, build_od_candidates
- CSV의 유량 컬럼을 자동 감지한다.
- passenger_car, bus, subway, rail, high_speed_rail, air, sea, freight 등 모드별 컬럼이 있으면 가중치를 적용해 total_flow를 만든다.
- freight 관련 컬럼은 weight 1.5가 적용된다.
- bus는 weight 1.2, subway는 weight 0.8이 적용된다.
- total_flow 또는 flow 컬럼만 있으면 weight 1.0으로 처리한다.
- total_flow가 0 이하인 행은 제외한다.
- 좌표가 유효하지 않은 행은 제외하거나 행정동 좌표 lookup으로 보완한다.

행정동 좌표 매칭 방식:
- 사용 파일: backend/app/services/od_candidate_generation.py
- 관련 함수: _load_coordinate_lookup, _row_to_record
- 기본 좌표 파일: data/processed/admin_dong_coordinates.csv
- OD CSV에 origin_admin_dong_code, destination_admin_dong_code가 있고 좌표가 없으면, admin_dong_code 기준으로 latitude, longitude를 찾아 채운다.
- 좌표 파일이 없거나 컬럼이 부족하면 lookup을 사용하지 않는다.
- 행정동 코드와 좌표 파일의 매칭률 상세 검증 로직은 현재 코드에서 확인 불가.

후보 정점 생성 방식:
- 사용 파일: backend/app/services/od_candidate_generation.py
- 관련 함수: _records_to_points, _weighted_kmeans_nodes, _merge_nearby_nodes, _prune_nodes
- 각 OD 행을 출발지 점과 도착지 점 2개로 분해한다.
- 각 점에는 해당 OD의 total_flow를 가중치로 부여한다.
- K-means 중심점을 후보 정점으로 만든다.
- 클러스터 중심점별 cluster_total_flow와 included_od_count를 계산한다.
- 가까운 정점은 merge_distance_km 3.0km 기준으로 병합한다.
- low_impact_prune_percent 기본값 20%로 수요가 낮은 정점을 제거한다.
- top_node_limit 기본값은 API에서 100으로 전달된다.

K-means 또는 클러스터링 사용 여부:
- 사용 여부: 사용함
- 사용 파일: backend/app/services/od_candidate_generation.py
- 관련 함수: _weighted_kmeans_nodes
- 라이브러리: sklearn.cluster.KMeans
- 기본 반복 K 값: 10, 20, 30, 50
- sample_weight에 OD 유량을 넣어 weighted K-means를 수행한다.

후보 연결쌍 생성 기준:
- 사용 파일: backend/app/services/od_candidate_generation.py
- 관련 함수: _generate_candidate_edges, _flush_edge_batch
- 각 OD의 출발지와 도착지를 가장 가까운 후보 정점에 매칭한다.
- 같은 정점으로 매칭된 OD는 연결쌍에서 제외한다.
- 정점 쌍별 estimated_flow를 합산한다.
- 직선거리 5km 미만 또는 100km 초과는 제외한다.
- min_estimated_flow가 주어지면 그보다 작은 연결쌍은 제외한다.
- estimated_flow 내림차순으로 정렬해 edge_limit만큼 반환한다.

기존 도로망 활용 방식:
- 사용 파일: backend/app/services/road_graph_routing.py, backend/app/services/road_network.py
- 관련 함수: nearest_road_node, build_road_graph_route, _query_corridor_edges, _find_road_path
- road_nodes에서 후보 정점과 가장 가까운 도로 노드를 찾는다.
- road_links를 corridor 범위로 조회해 그래프를 구성한다.
- road_rank에 따라 비용 가중치를 낮춘다.
- 기존 터널로 추정되는 도로 링크는 비용을 0.9배로 낮춘다.
- 출발/도착 후보 정점에서 도로망까지는 connector segment로 만든다.

A* 알고리즘 사용 위치와 역할:
- 기존 도로망 A*: backend/app/services/road_graph_routing.py의 _find_road_path
- DEM 비용격자 A*: backend/app/services/pathfinding.py의 find_least_cost_path
- 기존 도로망 A*는 road_links 그래프에서 기존 도로 기준 경로를 찾는다.
- DEM 비용격자 A*는 경사도·회피지역·기존도로 접근성 등을 반영한 격자에서 신규/우회 경로를 찾는다.

DEM 기반 신규 도로/터널 경로 탐색 방식:
- 사용 파일: backend/app/services/cost_grid.py, backend/app/services/pathfinding.py, backend/app/services/candidate_route_pipeline.py
- 관련 함수: generate_dem_route_grid, build_cost_grid, find_least_cost_path, _dem_link
- 출발·도착 좌표 주변에 DEM 기반 비용 격자를 생성한다.
- PostGIS의 dem_elevation 테이블에서 고도를 조회한다.
- 고도가 없는 셀은 통과 불가 비용으로 처리한다.
- 하천, 건물, 보호구역, 도시지역, 기존 도로 접근성 레이어가 있으면 비용을 조정한다.
- A*가 최저비용 격자 경로를 찾는다.
- 첫 시도 실패 시 buffer_multiplier를 2배로 늘려 재시도한다.

경사도 계산 방식:
- 사용 파일: backend/app/services/cost_grid.py
- 관련 함수: _calculate_slopes, _slope_cost
- 각 격자 셀과 상하좌우 이웃 셀의 고도 차이를 cell_size_m으로 나눠 각도를 계산한다.
- 해당 셀의 slope_degrees는 이웃 방향 경사 중 최대값이다.
- 경사도별 격자 비용:
  5도 미만: 1.0
  10도 미만: 2.0
  15도 미만: 4.0
  25도 미만: 10.0
  그 이상: 30.0

터널 판단 기준:
- 사용 파일: backend/app/services/route_segments.py, backend/app/services/geotechnical_model.py
- 관련 함수: _initial_type, _atom_with_decision, evaluate_tunnel_decision
- 초기 판단은 경사도 25도 이상 또는 주변 기복 80m 이상이면서 비용이 높은 구간이면 tunnel 후보로 본다.
- 이후 geotechnical_model의 터널 판단 로직을 적용한다.
- 토피고 또는 암반등급 정보가 없으면 fallback_slope_based_tunnel_logic으로 경사도 기반 판단을 사용한다.
- 최종 터널이 되려면 원래 tunnel 후보이고, 도로 종단경사 8% 이상이며, 터널 비용이 지상도로 비용의 1.10배 이하여야 한다.
- 연속 터널 길이가 0.8km 미만이면 신규 지상도로로 낮춘다.
- 터널 사이의 짧은 비터널 gap이 0.2km 미만이면 터널로 병합한다.

도로/터널/교량 segment 분류 방식:
- 사용 파일: backend/app/services/route_segments.py, backend/app/services/road_graph_routing.py
- 관련 함수: classify_route_segments, _append_segment, _merge_existing_road_segments
- existing_road: 기존 도로망 또는 기존 도로 중심선 인근으로 판정된 구간
- connector: 후보 정점에서 기존 도로망까지 연결하는 접속 구간
- new_surface_road: DEM 기반 신규 지상도로 구간
- tunnel: DEM 경로 중 터널로 최종 판정된 구간
- river_crossing 구간은 bridge가 tunnel보다 우선하며 결과 contract와 지도에 길이·개수가 포함된다. 다만 전용 교량비 모델은 없어 후보 스크리닝 비용에서는 도로 단가를 적용하고, 상세 보고서에서는 별도 검토 경고를 남긴다.

후보 노선별 거리, 시간, 비용, 편익 계산 방식:
- 거리: segment 길이를 합산하거나 DEM cell 간 거리를 합산한다.
- 시간: 기존 도로 60km/h, 신규도로 80km/h, connector 40km/h, 터널 80km/h로 계산한다.
- 비용: 신규 지상도로, connector, 터널, 토지보상비를 합산한다.
- 편익: 기존 도로 기준 경로 대비 거리 절감과 시간 절감을 estimated_flow 및 diversion_rate에 곱해 계산한다.

estimated_flow 사용 방식:
- 후보 연결쌍의 수요 크기를 나타낸다.
- edge 정렬 기준으로 사용된다.
- 편익 계산에서 estimated_flow에 diversion_rate를 곱해 diverted_flow를 만든다.
- annual_time_benefit과 annual_distance_benefit은 diverted_flow 기준으로 계산한다.
- 참고로 annual_benefit_before_diversion은 estimated_flow 전체를 이용한 값으로 별도 계산되지만, 최종 total_benefit은 diverted_flow 기준이다.

현재 전환율 또는 교통량 배분 모델 여부:
- 전환율 모델은 있음.
- 사용 파일: backend/app/services/cost_model.py
- 관련 함수: calculate_diversion_rate
- 절감률 기반 단순 전환율이다.
- 실제 교통 배분, 네트워크 assignment, 혼잡 반영, 로짓 모델은 현재 코드에서 확인 불가.

후보 노선 유효성 판단 기준:
- OD 후보 연결쌍은 직선거리 5~100km 범위여야 한다.
- 신규 후보 노선은 DEM 경로 탐색에 성공해야 status가 success가 된다.
- meaningful improvement 기준은 거리 절감률 5% 이상 또는 시간 절감률 5% 이상이다.
- 개선 기준을 충족하지 못하면 후보 점수에 40점 패널티가 붙는다.
- 전환율이 0이면 후보 점수는 최대 10점으로 제한된다.

후보 노선 점수화 또는 순위화 방식:
- 사용 파일: backend/app/services/cost_model.py, backend/app/services/route_economics.py
- 관련 함수: evaluate_candidate_against_baseline, rank_candidate_routes
- 개별 candidate_score는 거리 절감, 시간 절감, 전환 통행량, B/C, 기존 도로 활용률을 가점으로 사용한다.
- 신규 건설 비율, 비용 부담, 터널 비율, 지형 위험, 너무 짧은 신규구간, 5% 개선 미달은 감점으로 사용한다.
- 기존 도로 기준 경로는 candidate_score 50점으로 설정되지만 ranked_routes에서는 제외된다.
- rank_candidate_routes는 성공한 신규/혼합 후보만 정렬한다.


4. 비용 계산 구조

접속 도로 건설비:
- 사용 여부: 사용함
- 계산 방식: connector_length_km × 184.0억원/km × 도시 유형 계수 × 설계속도 계수
- 관련 파일: backend/app/services/cost_model.py
- 관련 함수: calculate_route_costs
- 발표자료용 설명: 후보 정점과 기존 도로망을 이어주는 짧은 접속 구간도 신규 건설비로 계산한다.

터널 건설비:
- 사용 여부: 사용함

- 관련 함수: calculate_route_costs, estimate_tunnel_segment_cost_eok, getGroundFactorByGeology
- 주요 단가: 2차로 NATM 터널 13,940천원/m, 3차로 22,959천원/m
- 발표자료용 설명: 터널은 길이와 암반 조건에 따라 비용이 크게 달라지므로, 지반 계수와 터널 길이 계수를 반영한다.

유지관리비:
- 사용 여부: 현재 코드에서 확인 불가
- 계산 방식: 현재 후보 노선 비용 계산식에는 유지관리비 항목이 별도로 없음
- 관련 파일: 현재 코드에서 확인 불가
- 관련 함수: 현재 코드에서 확인 불가
- 발표자료용 설명: 현재 MVP는 초기 건설비와 토지보상비 중심이며, 유지관리비는 향후 확장 항목이다.

토지 보상비:

- 계산 방식: 신규 건설 구간 geometry를 도로 폭 20m 기준으로 buffer 처리한 뒤, 교차 필지 면적 × 공시지가 또는 추정지가 × 토지유형별 보정계수로 계산한다. 결과는 원 단위에서 억원 단위로 변환되어 총비용에 합산된다.
- 관련 파일: backend/app/services/land_compensation.py, backend/app/services/vworld_parcel_repository.py, backend/app/services/candidate_route_pipeline.py
- 관련 함수: estimate_land_compensation, _apply_land_compensation, get_default_parcel_repository
- 발표자료용 설명: 신규로 건설해야 하는 구간만 대상으로 편입 토지 면적과 가격을 추정해 보상비를 더한다.

경사도에 따른 비용 가중치:

- 발표자료용 설명: 경사가 급한 지역은 길을 내기 어렵다고 보고, 경로 탐색 단계에서 우회되도록 비용을 높인다.

지형/DEM 기반 비용 가중치:
- 사용 여부: 사용함
- 계산 방식: DEM 고도와 경사도, 하천, 건물/보호구역/도시지역, 기존 도로 접근성을 격자 비용에 반영한다.

현재 MVP 계산식과 향후 확장 가능 부분:

- 계산 방식: 신규도로비 + 접속도로비 + 터널비 + 토지보상비 + 예비비 형태이다. 도로 예비비 25%, 터널 예비비 35%가 적용된다.
- 관련 파일: backend/app/services/cost_model.py, backend/app/services/route_mvp_config.py
- 관련 함수: calculate_route_costs

5. 편익 계산 구조

시간 절감 편익:
- 사용 여부: 사용함
- 계산 방식: 기존 도로 기준 시간 - 후보 노선 시간 = time_saving. time_saving × diverted_flow × 365일 × 시간가치 20,000원/시간 ÷ 100,000,000으로 연간 억원 단위 편익을 계산한다.
- 관련 파일: backend/app/services/cost_model.py
- 관련 함수: evaluate_candidate_against_baseline
- 발표자료용 설명: 새 노선을 이용해 줄어드는 이동시간을 돈으로 환산해 연간 편익으로 계산한다.

거리 절감 편익:
- 사용 여부: 사용함
- 계산 방식: 기존 도로 기준 길이 - 후보 노선 길이 = distance_saving. distance_saving × diverted_flow × 365일 × 차량운행비 200원/km ÷ 100,000,000으로 연간 억원 단위 편익을 계산한다.
- 관련 파일: backend/app/services/cost_model.py
- 관련 함수: evaluate_candidate_against_baseline
- 발표자료용 설명: 짧아진 이동거리만큼 차량 운행비가 줄어든다고 보고 편익을 계산한다.

물류비 절감:
- 사용 여부: 별도 물류비 모델은 현재 코드에서 확인 불가
- 계산 방식: freight_flow는 OD 유량 산정 시 weight 1.5로 더 크게 반영된다. 그러나 화물 시간가치, 화물차 운행비, 물류비 절감식을 별도로 나누어 계산하는 구조는 현재 코드에서 확인 불가.
- 관련 파일: backend/app/services/od_candidate_generation.py, backend/app/services/cost_model.py
- 관련 함수: _detect_flow_columns, _row_total_flow, evaluate_candidate_against_baseline
- 발표자료용 설명: 화물 수요는 후보지 선정 단계에서 더 크게 반영되지만, 최종 편익은 시간·거리 절감 중심의 단순 모델이다.

통행량 또는 estimated_flow 반영 방식:
- 사용 여부: 사용함
- 계산 방식: estimated_flow × diversion_rate = diverted_flow. 최종 annual_benefit과 total_benefit은 diverted_flow를 기준으로 한다.
- 관련 파일: backend/app/services/cost_model.py
- 관련 함수: evaluate_candidate_against_baseline
- 발표자료용 설명: 모든 수요가 새 노선으로 이동한다고 보지 않고, 절감 효과가 클수록 일부 수요가 전환된다고 본다.

5% 개선 기준:
- 사용 여부: 사용함
- 계산 방식: 거리 절감률 5% 이상 또는 시간 절감률 5% 이상이면 meaningful improvement로 본다. 5% 미만이면 candidate_score에 40점 패널티가 적용된다.
- 관련 파일: backend/app/services/route_mvp_config.py, backend/app/services/cost_model.py
- 관련 함수: evaluate_candidate_against_baseline
- 발표자료용 설명: 새 노선이 실제로 의미 있으려면 기존 경로보다 최소 5% 이상 거리 또는 시간을 줄여야 한다.

신규 노선 이용 전환율:
- 사용 여부: 사용함
- 계산 방식: saving_score = 거리절감률 × 0.4 + 시간절감률 × 0.6. saving_score가 0.05 미만이면 diversion_rate는 0이다. 그 이상이면 saving_score × 3.0을 적용하되 최대 0.70으로 제한한다.
- 관련 파일: backend/app/services/cost_model.py, backend/app/services/route_mvp_config.py
- 관련 함수: calculate_diversion_rate
- 발표자료용 설명: 시간과 거리가 많이 줄어들수록 더 많은 차량이 새 노선을 선택한다고 보는 단순 전환율 모델이다.

현재는 estimated_flow 전체가 신규 노선을 이용한다고 가정하는지 여부:
- 사용 여부: 최종 편익에서는 전체 이용을 가정하지 않음
- 계산 방식: annual_benefit_before_diversion과 total_benefit_before_diversion은 estimated_flow 전체 기준으로 별도 계산된다. 하지만 최종 annual_benefit, total_benefit, B/C는 diverted_flow 기준이다.
- 관련 파일: backend/app/services/cost_model.py
- 관련 함수: evaluate_candidate_against_baseline
- 발표자료용 설명: 참고용으로 전체 수요가 이용했을 때의 편익도 계산하지만, 순위화에는 전환율을 반영한 보수적 편익이 사용된다.


6. 기존 도로 활용 구조

기존 도로망 A* 탐색이 우선인지:
- 후보 노선 계산에서 apply_optional_layers가 True이면 기존 도로 기준 경로를 먼저 만든다.
- 관련 파일: backend/app/services/candidate_route_pipeline.py
- 관련 함수: evaluate_candidate_edge_candidates, _baseline_candidate
- 기본 API /api/candidate-routes는 apply_optional_layers 기본값 True로 build_candidate_routes를 호출하므로, 실제 웹 흐름에서는 기존 도로 기준 경로가 먼저 계산된다.

신규 도로는 기존 도로 연결 보조구간인지:
- 두 가지가 모두 있다.
- new_direct는 후보 정점 간 DEM 기반 직접 신규 경로이다.
- hybrid_new_existing, bypass_improvement, tunnel_shortcut은 기존 도로 기준 경로의 일부를 DEM 신규 링크로 대체하는 구조이다.
- 기존 도로 기반 혼합 후보에서는 기존 도로가 앞뒤 접속·보조 구간으로 활용된다.

기존 도로 탐색 실패 시 DEM 기반 신규 노선 탐색이 실행되는 구조인지:
- 그렇다.
- 기존 도로 baseline 생성 실패는 경고 로그만 남기고, new_direct DEM 후보는 별도로 실행된다.
- baseline이 없으면 hybrid_new_existing, bypass_improvement, tunnel_shortcut은 생성되지 않는다.

경제성 계산에서 기존 도로 구간의 건설비가 제외되는지:
- 제외된다.
- calculate_route_costs는 new_surface_road_length_km, connector_length_km, tunnel_length_km, land_compensation_cost_eok만 비용으로 계산한다.
- existing_road_length_km은 비용 계산에 들어가지 않는다.
- 기존 도로 baseline은 신규 도로와 터널 길이가 0이고 connector 비용만 계산된다.

현재 구조가 신규 노선 후보 생성보다 기존 도로 최적경로 탐색에 가까운지 여부:
- 현재 구조는 “기존 도로 기준 경로 + DEM 신규/우회 후보 비교”에 가깝다.
- 신규 직접 노선도 계산하지만, 혼합 후보는 기존 도로 경로의 일부 구간을 DEM 신규 링크로 대체하는 방식이다.
- 따라서 완전한 신규 도로망 설계 시스템이라기보다, 기존 도로망을 기준선으로 삼아 개선 가능 구간을 찾는 초기 검토 도구로 보는 것이 정확하다.

9. 발표자료용 요약

[한 장 요약]

- 서비스 목적:
OD 교통수요 데이터를 기반으로 신규 도로·터널 후보 노선을 자동으로 찾고, 기존 도로 대비 개선 효과와 사업비를 비교하는 초기 검토 서비스이다.

- 입력 데이터:
OD CSV, 행정동 좌표, 도로망 DB, DEM 고도 데이터, 선택 시나리오 OD, 지역 필터, 선택적으로 하천·건물·보호구역·필지 데이터.

- 계산 과정:
OD 데이터를 수요 중심 후보 정점으로 압축하고, 수요가 큰 정점 쌍을 후보 연결쌍으로 만든다. 각 연결쌍에 대해 기존 도로망 A* 기준 경로와 DEM 비용격자 A* 신규/우회 경로를 계산한다. 이후 구간을 기존 도로·접속 도로·신규 도로·터널로 나누고, 비용과 편익을 계산해 순위를 매긴다.

- 출력 결과:
후보 노선 지도, 구간별 색상 표시, 총 길이, 신규 도로 길이, 터널 길이, 예상 수요, 전환율, 비용, 편익, B/C, 후보 점수, 순위.

- 차별점:
단순 직선 후보가 아니라 기존 도로망과 DEM 지형을 함께 사용해 “기존 도로 기준 경로 대비 어떤 신규/우회 노선이 더 나은지”를 비교한다.

- 현재 한계:
정밀 교통 배분 모델은 없다. 교량비와 유지관리비는 계산하지 않는다. 지질·암반·토피고 모델은 일부 구현되어 있으나 메인 DEM 비용격자에 실제 연결되는 흐름은 현재 코드에서 확인 불가. 프론트엔드 문자열 일부는 인코딩이 깨져 보인다.

- 향후 개선 방향:
실제 교통 배분 모델, 교량비, 유지관리비, 정밀 지질/토피고 반영, 공법별 단가, 환경영향 비용, 민감도 분석, 보고서 자동 생성 고도화를 추가할 수 있다.

[슬라이드 구성안]

1. 문제 정의
- 도로·터널 후보지는 수요, 지형, 기존 도로망, 비용을 함께 봐야 한다.
- 수작업 검토는 시간이 오래 걸리고 초기 후보를 빠르게 비교하기 어렵다.

2. 서비스 개요
- ONESTEP은 OD 교통수요를 기반으로 신규 도로·터널 후보를 자동 산정한다.
- 사용자는 OD 데이터와 지역 조건을 넣고, 후보 노선 지도와 순위를 받는다.

3. 입력 데이터 구조
- OD 데이터: 출발지, 도착지, 수단별 유량
- 행정동 좌표: 코드 기반 좌표 보완
- 도로망: road_nodes, road_links
- DEM: 고도와 경사도
- 선택 데이터: 하천, 건물, 보호구역, 필지

4. 후보 정점과 연결쌍 생성
- OD 출발지·도착지를 유량 가중 점으로 변환
- weighted K-means로 수요 중심점 생성
- 가까운 중심점 병합
- 수요가 큰 정점 쌍을 후보 연결쌍으로 선정

5. 노선 탐색 방식
- 기존 도로망 A*로 기준 경로 계산
- DEM 비용격자 A*로 신규 직접 경로와 우회 경로 계산
- 기존 도로, 접속 도로, 신규 지상도로, 터널 segment로 분류

6. 비용·편익 계산
- 비용: 신규도로비, 접속도로비, 터널비, 토지보상비, 예비비
- 편익: 시간 절감, 거리 절감, 전환 통행량
- 5% 이상 개선 기준과 전환율을 반영해 후보 점수 계산

7. 앱 작동 화면
- 좌측 조건 설정
- 중앙 지도 표시
- 우측 후보 연결쌍 및 노선 순위 패널
- segment 색상으로 기존도로, 신규도로, 터널 표시

8. 한계 및 고도화 방향
- 현재 MVP는 초기 검토용 개략 모델
- 교통 배분, 교량비, 유지관리비, 정밀 지질 반영 필요
- 향후 사업성 보고서와 시나리오 비교 기능으로 확장 가능


10. 현재 구현과 추후 계획 구분

현재 구현됨:
- OD CSV 처리
- 행정동 좌표 lookup
- weighted K-means 후보 정점 생성
- 후보 연결쌍 생성
- 기존 도로망 A* 기준 경로
- DEM 비용격자 A* 신규 경로
- 기존도로/connector/신규 지상도로/터널 segment 분류
- 신규도로·터널·접속도로 비용 계산
- 토지보상비 계산 구조와 VWorld 필지 저장소 연결
- 시간·거리 절감 편익
- 전환율 기반 편익
- 후보 점수화와 순위화
- 지도 표시용 geometry 응답

현재 코드에서 확인 불가:
- 정밀 교통 배분 모델
- 혼잡도 반영
- 유지관리비 계산
- 교량 segment 및 교량비 계산
- 실제 물류비 별도 편익 모델
- 환경영향 비용 정량화
- 메인 DEM 비용격자에 지질 샘플링 결과를 주입하는 연결
- 암반등급·토피고가 실제 운영 데이터에서 항상 반영되는지 여부
