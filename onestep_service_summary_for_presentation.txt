0. 문서 작성 원칙
======================================================================

이 문서는 ONESTEP 프로젝트의 실제 소스 코드, API 스키마, 프론트엔드 요청 흐름,
데이터 적재 스크립트, 현재 로컬 PostGIS 연결 상태를 확인하여 작성했다.

표시 기준은 다음과 같다.

- [구현 완료] 현재 주 실행 흐름에서 호출되며 결과에 반영되는 기능
- [일부 구현] 코드 또는 화면은 있으나 계산 범위가 제한되거나 주 실행 흐름과 불일치하는 기능
- [현재 미연결] 처리 코드는 있으나 현재 로컬 데이터베이스 또는 화면 흐름에 연결되지 않은 기능
- [현재 구현되지 않음] 해당 기능을 수행하는 코드를 확인할 수 없는 기능
- [발표 주의] 실제 구현보다 확대하여 설명하면 안 되는 부분

현재 로컬 DB 테이블 확인 결과는 이 문서를 작성한 실행 환경의 상태다. 다른 PC나 배포
환경에서는 데이터 적재 여부에 따라 달라질 수 있다.


======================================================================
1. 서비스 개요
======================================================================

1.1 서비스 이름과 목적

서비스 이름은 ONESTEP이다.

한 줄 정의:
교통 OD 수요를 출발점으로 기존 도로망, DEM 지형, 건물·도시 회피 정보 등을 결합하여
도로·터널 예비 후보 노선을 만들고, 공사비·토지보상비·시간 및 거리 절감 편익을 비교하는
사업 초기 검토용 MVP 서비스이다.

확인 파일:
- frontend/src/App.jsx
- backend/app/services/od_candidate_generation.py
- backend/app/services/candidate_route_pipeline.py
- backend/app/services/cost_grid.py
- backend/app/services/cost_model.py

1.2 사용자가 해결하려는 문제

코드에서 확인되는 문제 정의는 다음과 같다.

1) 전국 또는 선택 지역의 많은 OD 데이터를 사람이 하나씩 검토하기 어렵다.
2) 수요가 집중되는 위치와 연결축을 자동으로 줄여서 후보군으로 만들 필요가 있다.
3) 단순 직선이 아니라 기존 도로망과 지형을 고려한 비교 노선이 필요하다.
4) 노선 길이만이 아니라 신설도로, 접속도로, 터널, 토지보상비와 예상 편익을 함께 비교할
   필요가 있다.
5) 후보를 지도, 순위, 예비 보고서 형태로 빠르게 검토할 필요가 있다.

1.3 기존 방식과의 차별점으로 설명할 수 있는 부분

[코드에서 확인]
- OD 수요를 가중치로 사용해 여러 K 값의 K-Means 중심점을 계산한다.
- 기존 도로망 A* 기준 노선과 DEM 비용격자 A* 신설 대안을 함께 만든다.
- 기존도로 기준선, 완전 신설형, 기존도로 일부 대체형, 우회 개선형을 같은 연결쌍에서
  비교한다.
- DEM 경사도뿐 아니라 데이터가 존재할 경우 도로 접근성, 하천, 건물·도시, 지질,
  단층·지질경계 정보를 비용격자 또는 구간 판단에 반영하도록 설계되어 있다.
- 노선별로 공사비, 토지보상비, 예상 편익, B/C, 순편익, 자체 후보 점수를 계산한다.

[발표 주의]
- 정밀 도로설계, 정밀 터널설계, 교통수요 배정, 환경영향평가를 수행하는 시스템은 아니다.
- 현재 결과는 “사업 초기 예비 후보 비교” 수준이다.
- 현재 DEM 격자는 기본 500m이며, 도로 선형과 종단을 설계도 수준으로 생성하지 않는다.
- 기존도로 판정에는 확인된 오분류 문제가 있다. 자세한 내용은 11장 한계점에 정리한다.

1.4 사용자가 입력하거나 선택하는 값

주 화면에서 확인되는 입력:

- 기본 OD 데이터 포함 또는 제외
- 사용자 CSV 업로드
- 화면에서 만든 추가 OD 시나리오 선택
- 시·도 계산 구역 선택
- 선택 지역 경계 여유거리 0~100km
- OD 유량 상위 필터: 전체, 상위 5%, 10%, 20%
- 영향도가 낮은 후보 정점 제거: 하위 20%, 하위 30%, 제거 없음
- 후보 연결쌍 상한: 20, 50, 100
- 샘플링: 사용 안 함, 5,000, 20,000, 50,000행
- 최종 노선 탐색 후보 수: 5, 10, 15, 20, 30, 50

시나리오 생성 화면의 입력:

- 시나리오 이름
- 지도 클릭 또는 카카오 장소 검색을 통한 출발지·도착지
- OD 이름
- 물동량
- 교통량

관련 화면:
- frontend/src/App.jsx / AnalysisMvpPage
- frontend/src/App.jsx / ScenarioCreatePage

별도 API에서 받을 수 있지만 현재 주 프론트엔드 화면에서 직접 사용하지 않는 입력:

- 수동 출발·도착 좌표와 도로·터널 단가
- 암반계수, 샘플 간격, 급경사 도로 할증
- 직접 입력한 건설비·연간 편익·운영비·할인율·운영연수

관련 API:
- POST /api/route-cost
- POST /api/evaluate-route
- POST /api/generate-route
- POST /api/analysis

1.5 시스템 출력

주 후보 분석 결과:

- OD 후보 정점 목록
- 후보 정점 연결쌍과 추정 수요
- 노선 후보 목록
- 구간별 유형: 기존도로, 접속도로, 신설 지상도로, 터널
- 노선별 길이와 구간별 길이
- 직접공사비, 예비비 포함 공사비, 토지보상비
- 거리 절감, 시간 절감, 전환률, 전환 수요
- 연간 시간 편익, 연간 거리 편익, 총 현재가치 편익
- B/C ratio, 순편익, 후보 점수, 순위
- 지도 선형과 후보 선택 포커스
- 처리 건수, A* 호출 수, 격자 수, 실행 시간, 경고
- 템플릿 기반 예비 후보 노선 검토 보고서와 Markdown 다운로드

결과 JSON 파일:
- data/processed/candidate_nodes.json
- data/processed/candidate_edges.json
- data/processed/candidate_routes.json
- data/processed/candidate_route_segments.json

2. 전체 앱 작동 흐름
======================================================================

2.1 사용자가 웹에서 사용하는 순서

1) 시작 화면
   - ONESTEP 소개 화면에서 “서비스 시작하기”를 누른다.
   - 파일: frontend/src/App.jsx / App

2) 필요하면 OD 시나리오 생성
   - 상단 “시나리오 생성” 화면으로 이동한다.
   - 지도 클릭 또는 카카오 장소 검색으로 출발지와 도착지를 선택한다.
   - OD 이름, 물동량, 교통량을 입력하여 여러 OD를 만든다.
   - 시나리오 저장 시 React 메모리 상태에 추가되고 분석 화면으로 돌아간다.
   - [발표 주의] 서버 DB나 브라우저 localStorage에 영구 저장하지 않는다. 새로고침하면
     사용자가 만든 시나리오는 사라진다.

3) OD 데이터 소스 선택
   - 프로젝트 기본 OD CSV, 업로드 CSV, 화면에서 만든 추가 시나리오를 선택한다.
   - 추가 시나리오는 프론트엔드가 임시 CSV로 만들어 백엔드에 보낸다.
   - 교통량은 passenger_car, 물동량은 freight 열로 전달된다.

4) 분석 범위와 필터 선택
   - 시·도, 경계 여유거리, 상위 OD 필터, 하위 정점 제거율, edge 수, 샘플 수를 정한다.

5) “후보 생성 실행”
   - 프론트엔드가 multipart/form-data로 POST /api/od-candidates를 호출한다.
   - 반환된 후보 정점과 후보 연결쌍을 지도와 우측 목록에 표시한다.

6) “도로·터널 후보 생성”
   - 반환받은 nodes와 edges를 JSON으로 POST /api/candidate-routes에 전달한다.
   - 백엔드는 각 연결쌍에 대해 기존도로 기준선과 신설·혼합 대안을 계산한다.

7) 지도 및 순위 검토
   - 지도에는 기존도로, 신설도로, 접속도로, 터널이 다른 색과 선 스타일로 표시된다.
   - 우측 카드에는 길이, 수요, 거리 절감, 구간별 길이, 공사비, 자체 점수가 표시된다.
   - 후보 카드를 클릭하면 해당 연결쌍 중심으로 지도가 이동한다.

8) 예비 보고서 보기
   - 후보의 “보고서 보기” 버튼을 누른다.
   - 프론트엔드가 현재 routes, segments, costs, ranked_routes를
     POST /api/candidate-routes/reports로 보낸다.
   - 백엔드의 TemplateReportGenerator가 템플릿 보고서를 생성한다.
   - 화면에서 보고서를 확인하고 Markdown 파일로 내려받을 수 있다.
   - [현재 구현되지 않음] PDF 직접 생성
   - [현재 구현되지 않음] ChatGPT 또는 다른 생성형 AI를 호출하는 보고서 생성

2.2 프론트엔드 → 백엔드 요청 흐름

주 실행 흐름:

React AnalysisMvpPage
  → POST /api/od-candidates
  → ODCandidateResult(nodes, edges, stats)
  → POST /api/candidate-routes
  → CandidateRouteResult(routes, segments, costs, ranked_routes, best_candidate)
  → POST /api/candidate-routes/reports
  → CandidateRouteReportsResult

프론트엔드 API 기본 주소:
- VITE_API_BASE_URL이 없으면 http://localhost:8000

지도:
- 카카오 Maps JavaScript SDK
- 키: VITE_KAKAO_MAP_JS_KEY

2.3 백엔드 계산 순서

후보 정점·연결쌍:

1) CSV 열 구조 탐지
2) 모드별 유량 가중합 계산
3) 좌표 또는 행정동 좌표 lookup 적용
4) 0 이하 수요, 좌표 누락, 선택 지역 밖 OD 제외
5) 필요 시 상위 유량 필터 또는 샘플 적용
6) 출발점과 도착점을 하나의 가중 포인트 집합으로 구성
7) K=10, 20, 30, 50의 weighted K-Means 반복
8) 3km 이내 중심점 병합
9) 하위 영향 정점 제거와 상위 정점 수 제한
10) 모든 유효 OD의 출발·도착을 가장 가까운 후보 정점에 배정
11) 정점쌍별 수요 합산
12) 거리 5~100km 조건과 최소 수요 조건 적용
13) estimated_flow가 높은 연결쌍부터 반환

노선 후보:

1) estimated_flow가 높은 edge부터 처리
2) 기존 도로망 A* 기준선 생성
3) DEM 비용격자 A* 완전 신설형 생성
4) 기존 기준선의 일부 구간을 DEM 링크로 교체한 혼합형과 우회 개선형 생성
5) 직접 신설형에서 터널이 확인될 때만 터널 단축형 추가 검토
6) DEM 셀 경로를 기존도로·신설도로·터널로 분류
7) 접속도로를 포함해 구간 목록 구성
8) 신설·접속 구간에 토지보상비 적용
9) 공사비 재계산
10) 기존도로 기준선 대비 거리·시간 절감과 전환률 계산
11) 편익, B/C, 순편익, 후보 점수 계산
12) 후보 점수와 전환 수요 기준으로 정렬
13) 지도·표·파일용 결과 반환

3. 후보 노선 생성 로직
======================================================================

3.1 OD 데이터 사용 방식

주 모듈:
- backend/app/services/od_candidate_generation.py

기본 OD 파일:
- data/processed/synthetic_admin_dong_od_by_mode.csv

기본 좌표 lookup:
- data/processed/admin_dong_coordinates.csv

지원하는 수요 그룹과 기본 가중치:
- 승용차(car): 1.0
- 버스(bus): 1.2
- 지하철(subway): 0.8
- 철도(rail): 1.0
- 고속철도: 1.0
- 항공: 1.0
- 해운: 1.0
- 화물(freight): 1.5

계산:
total_flow = 각 모드 값 × 해당 모드 가중치의 합

지원 모드 열이 없으면 total_flow, flow, traffic 등의 단일 열을 찾는다.

[발표 주의]
- 서로 단위가 다를 수 있는 여객 통행량과 화물 물동량을 가중합한다.
- 가중치는 코드 상수이며 수요모형으로 보정된 계수가 아니다.

3.2 OD 필터와 좌표 처리

- total_flow가 0 이하인 행은 제외한다.
- CSV에 좌표가 없으면 행정동 코드를 admin_dong_coordinates.csv와 연결한다.
- 선택 지역 필터를 쓰면 출발점과 도착점이 모두 선택 지역 또는 경계 buffer 안에 있어야 한다.
- 시·도 범위는 실제 행정경계 polygon이 아니라 WGS84 bounding box이다.
  파일: backend/app/config/regions.py, backend/app/services/region_filter.py
- 상위 5%, 10%, 20% 옵션은 clustering 입력을 유량 상위 행으로 제한한다.

[구현상 차이]
- 상위 유량 필터는 K-Means용 행을 제한하지만, edge estimated_flow 집계는 다시 전체 유효
  원본 OD를 순회한다. 따라서 “클러스터는 상위 OD 기준, 연결쌍 수요는 전체 OD 기준”이다.
- sample_size를 쓰면 clustering은 reservoir sampling을 사용하지만 edge 집계는 파일 앞에서
  sample_size개 유효 행까지만 읽는다. 두 단계의 표본 방식이 일치하지 않는다.

3.3 OD 클러스터링과 후보 정점

함수:
- _records_to_points
- _weighted_kmeans_nodes
- _merge_nearby_nodes
- _prune_nodes

과정:

1) 각 OD의 출발점과 도착점을 모두 EPSG:5179 평면좌표로 변환한다.
2) 출발점과 도착점 각각에 동일한 total_flow 가중치를 준다.
3) K=10, 20, 30, 50으로 K-Means를 반복한다.
4) 각 반복에서 cluster_total_flow와 포함 포인트 수를 계산한다.
5) 모든 K 실행에서 나온 중심점 중 3km 이내 중심점을 합친다.
6) 병합 위치는 누적 cluster_total_flow 가중평균으로 계산한다.
7) flow가 낮은 하위 20%가 기본 제거 대상이다.
8) 최종적으로 최대 100개 정점을 남긴다.

3.4 후보 연결쌍 생성

함수:
- _generate_candidate_edges
- _flush_edge_batch

과정:

1) 각 OD 출발점과 도착점을 가장 가까운 최종 후보 정점에 배정한다.
2) 출발·도착이 같은 정점이면 제외한다.
3) 정점쌍 방향을 제거하고 하나의 무방향 pair로 합친다.
4) pair별 total_flow와 OD 행 수를 합산한다.
5) 직선거리 5km 미만 또는 100km 초과 pair를 제외한다.
6) 최소 estimated_flow 옵션이 있으면 적용한다.
7) estimated_flow 내림차순으로 edge_limit만큼 선택한다.

4.2 초기 구간 분류

연속된 DEM 셀 두 개를 하나의 원자 구간으로 보고 다음 순서로 판정한다.

1) 양 끝 셀이 모두 기존 도로 50m 이내
   → existing_road

2) 최대 셀 경사 25도 이상
   → tunnel 초기 후보

3) 주변 7개 셀 범위의 고저차가 80m 이상이고 셀 비용이 5 이상

4) 나머지
   → new_surface_road

경로 종단경사:
abs(끝점 고도 - 시작점 고도) / 구간 길이 × 100

4.3 터널 세부 판단

geotechnical_model.py에서 지원하는 입력:

- 추정 암반등급 또는 암반등급

- 지상도로 추정비
- 터널 추정비

암반등급별 점수와 지반계수:
- I: 암반점수 25, 지반계수 0.75
- II: 20, 0.90
- III: 15, 1.10
- IV: -5, 1.50
- V: -40, 2.50
- unknown: 0, 1.30

토피고 점수:
- 20m 미만: -40
- 20~50m: 10
- 50m 이상: 25
- 값 없음: 0

종단경사 점수:
- 8~12%: 20
- 12% 이상: 40

기타:
- 국지 고저차 80m 이상: +15
- 암반 V: 추가 페널티 20
- 토피고 20m 미만: 추가 페널티 20

- 30~60: 터널비가 지상도로비의 1.10배 이하일 때 tunnel
- 토피고 20m 미만: 지상 또는 개착식 검토 우선
- 암반 V이고 터널비가 높으면 회피 또는 지상도로 우선

최종 route_segments.py의 추가 조건:
- 초기 유형이 tunnel이어야 한다.
- 종단경사가 8% 이상이어야 한다.
- 터널비가 지상도로비의 1.10배 이하여야 한다.
- 위 조건을 모두 만족해야 최종 tunnel로 남는다.

연속성 후처리:
- 연속 터널 길이가 0.8km 미만이면 신설 지상도로로 하향한다.
- 두 터널 사이 비터널 간격이 0.2km 미만이면 터널로 합친다.
- 간격에 existing_road가 있으면 터널로 합치지 않는다.

5.1 단위

5.2 신설 일반도로와 접속도로

기본 도로 단가:
- 184억원/km

공식:
- 신설도로 직접비 = 신설도로 길이(km) × 184 × 도시계수 × 설계속도계수
- 접속도로 직접비 = 접속도로 길이(km) × 184 × 도시계수 × 설계속도계수
- 예비비 포함 도로비 = 직접비 × 1.25

도시계수:
- rural 1.00
- semi_urban 1.10
- urban 1.20

설계속도계수:
- 80km/h 1.00
- 100km/h 1.05

[현재 주 파이프라인]
- urban_type을 전달하지 않아 기본 rural 1.00을 사용한다.
- design_speed도 전달하지 않아 기본 80km/h를 사용한다.
- 기존도로 구간 자체에는 신규 건설비를 부과하지 않는다.

5.3 터널비

기본 NATM 단가:
- 2차로: 13,940천원/m
- 3차로: 22,959천원/m
- 주 파이프라인 기본은 2차로

공식:
터널 직접비 =
  터널 길이(m)
  × (천원/m 단가 ÷ 100,000)
  × 단면계수
  × 지반계수
  × 길이계수
  × 부대계수

기본:
- 단면계수 1.00
- 부대계수 1.05
- 지질 미확인 기본 지반계수 1.30
- 예비비 포함 터널비 = 터널 직접비 × 1.35

길이계수:
- 400m 이하 0.994
- 800m 이하 0.998
- 1,000m 이하 1.000
- 1,200m 이하 1.006
- 2,000m 이하 1.021
- 4,000m 이하 1.062
- 4,000m 초과 1.08

[발표 주의]
- 구간별 터널 판단에서는 암반등급별 비용을 계산할 수 있다.
- 그러나 후보 총공사비를 다시 계산할 때 calculate_route_costs에 geology_data를 전달하지
  않는다. 따라서 현재 최종 터널 총비용은 구간별 암반등급이 아니라 기본 지반계수 1.30을
  사용한다.

5.4 총 초기비용

- 직접비 = 신설도로 직접비 + 접속도로 직접비 + 터널 직접비 + 토지보상비
- 예비비 포함 비용(total_screen_cost)
  = 신설도로 직접비×1.25
  + 접속도로 직접비×1.25
  + 터널 직접비×1.35
  + 토지보상비

5.5 토지보상비

[구현 완료, 외부 VWorld 서비스 필요]

주 파일:
- backend/app/services/vworld_parcel_repository.py
- backend/app/services/land_compensation.py
- backend/app/services/candidate_route_pipeline.py / _apply_land_compensation

대상:
- connector
- new_surface_road
- 기존도로와 터널 본선은 직접 대상에서 제외

기본 도로 폭:
- 20m

처리:

1) 신설·접속 구간을 폭 20m로 buffer 처리한다.
2) VWorld WFS lp_pa_cbnd_bubun 레이어에서 PNU, 필지 polygon, 지목, jiga를 조회한다.
3) 1,000건을 넘는 영역은 BBOX를 재귀 분할한다.
4) PNU별 최신 gosi_year/gosi_month 값을 선택한다.
5) 필지와 도로 buffer의 실제 교차면적을 계산한다.
6) 필지별 보상비를 계산한다.

필지별 공식:
편입면적(m²) × 결정 공시지가(원/m²) × 토지유형 보정계수

보정계수:
- 임야 1.2
- 농지 1.4
- 주거 1.8
- 상업·공업 2.0
- 미분류 1.5

공시지가 누락 시 순서:
1) 2km 이내 거리 역수 가중 KNN, 최대 5개
2) 5km 이내 KNN
3) 같은 법정동 평균
4) 같은 시군구 평균
5) 주변 전체 중앙값
6) 모두 없으면 missing 처리하고 해당 필지 비용 0

[코드상 주의]
- DEFAULT_PRICE_PER_M2=100,000 상수가 있으나 최종 missing fallback에서 실제 기본가격으로
  사용하지 않는다.
- VWorld 키 또는 응답 오류가 나면 토지보상비가 0원이 되고 warnings에 이유가 기록된다.
- 별도 vworld_land_price.py의 법정동 평균 API는 API endpoint로 존재하지만 주 후보
  보상비 흐름은 VWorld parcel WFS의 jiga를 직접 사용한다.
- 법정동코드 CSV는 가격 데이터가 아니라 법정동 이름과 코드 매핑표다.

6. 편익 계산 구조
======================================================================

주 파일:

6.1 기준 경로

- 기존도로 기준선이 생성되면 그 경로 길이와 구간별 시간을 기준으로 사용한다.
- 기존도로 기준선이 없으면 직선거리 × 1.3을 가정 기존 경로 길이로 사용한다.

6.2 구간별 가정속도

- 기존도로: 60km/h
- 접속도로: 40km/h
- 신설 지상도로: 80km/h
- 터널: 80km/h

혼잡, 신호, 용량, 차종별 속도, 시간대별 속도는 반영하지 않는다.

6.3 거리·시간 절감

- 거리 절감 = max(0, 기준 경로 길이 - 후보 경로 길이)
- 시간 절감 = max(0, 기준 통행시간 - 후보 통행시간)
- 음수 절감은 모두 0으로 처리한다.

6.4 estimated_flow 사용

estimated_flow는 후보 edge에 배정된 weighted total_flow의 합이다.

사용 위치:
- 전환 전 시간 편익
- 전환 전 거리 편익
- 전환률을 곱한 diverted_flow
- 전환 후 시간·거리 편익
- 자체 후보 점수의 수요 항목

[발표 주의]
- estimated_flow는 교통배정모형으로 얻은 신규 노선 실제 이용량이 아니다.
- 여객과 화물 가중합이므로 단위를 단순 “대/일”로 단정하기 어렵다.

6.5 전환률

[구현 완료]

현재 100% 전환 가정이 아니다.

saving_score =
  거리 절감률 × 0.4
  + 시간 절감률 × 0.6

- saving_score < 0.05: 전환률 0
- 그 이상: 전환률 = min(0.70, saving_score × 3.0)
- 최대 전환률 70%
- diverted_flow = estimated_flow × diversion_rate

6.6 시간 절감 편익

가정 시간가치:
- 20,000원/시간

연간 시간 편익(억원/년) =
시간 절감(시간/통행)
× diverted_flow
× 365
× 20,000
÷ 100,000,000

6.7 거리 절감 편익

가정 차량운행비:
- 200원/km

연간 거리 편익(억원/년) =
거리 절감(km/통행)
× diverted_flow
× 365
× 200
÷ 100,000,000

6.8 총 편익 현재가치

- 분석기간 30년
- 할인율 4.5%
- 연간 편익은 매년 동일하다고 가정
- 총편익 = 연간 편익 × 연금현가계수

반영하지 않는 항목:

- [현재 구현되지 않음] 연도별 교통량 증가율
- [현재 구현되지 않음] 램프업 또는 개통 초기 전환률 변화
- [현재 구현되지 않음] 잔존가치
- [현재 구현되지 않음] 사고 절감 편익
- [현재 구현되지 않음] 환경 편익
- [현재 구현되지 않음] 신뢰성 편익
- [현재 구현되지 않음] 혼잡 완화 네트워크 편익
- [현재 구현되지 않음] 별도 관광 편익
- [현재 구현되지 않음] 별도 산업 유발 편익
- [현재 구현되지 않음] 화물 시간가치·차량운행비를 분리한 물류비 절감

물동량은 수요 가중치에 1.5배로 포함되지만, 편익 단계에서는 별도 화물모형 없이 다른
estimated_flow와 같은 시간가치·km 비용을 사용한다.

7. 경제성 평가와 순위화
======================================================================

7.1 후보 노선 B/C

주 파일:
- backend/app/services/cost_model.py

- 비용 = total_screen_cost
- 편익 = 30년 시간·거리 편익 현재가치
- B/C = 총편익 / total_screen_cost
- 순편익 = 총편익 - total_screen_cost

[발표 주의]
- 유지관리비는 이 B/C 비용에 포함되지 않는다.
- total_screen_cost는 신설·접속·터널 예비비와 토지보상비를 포함한다.
- 기존도로 기준선은 비교 기준이며 신규 건설 후보 순위에서는 제외된다.

7.2 자체 후보 점수

신규 후보 점수는 0~100 범위로 제한된다.

가점:
- 거리 개선: 최대 30점
- 시간 개선: 최대 30점
- 전환 수요: 최대 10점
- B/C: 최대 15점
- 기존도로 활용률: 최대 20점

감점:
- 신설 구간 비율: 최대 25점
- 비용 부담: 최대 20점
- 신설 구간 중 터널 비율: 최대 15점
- 최대 경사 기반 지형위험: 최대 10점
- 신설도로+터널이 0.5km 미만인 사실상 접속도로 후보: 20점
- 거리 또는 시간 개선률이 둘 다 5% 미만: 40점

추가:
- 전환률이 0이면 최종 점수를 최대 10점으로 제한한다.
- 기존도로 기준선의 점수는 고정 50점이다.

7.3 최종 순위

route_economics.py는 총편익과 비용의 min-max 점수도 계산하지만, 실제 route row에
candidate_score가 항상 있으므로 최종 economic_score는 candidate_score를 사용한다.

정렬:
1) economic_score 내림차순
2) 동점이면 diverted_flow 내림차순

[발표 주의]
- 화면의 “MVP 예비 경제성 점수”는 B/C 그 자체가 아니라 여러 가점·감점을 합친 휴리스틱
  점수다.
- best_candidate는 반환 rows 전체에서 candidate_score 최대를 찾기 때문에 고정 50점의
  existing_baseline이 선택될 수 있다.
- ranked_routes는 existing_baseline을 제외한다. 따라서 best_candidate와 ranked_routes
  1위가 다를 가능성이 있다.

7.4 별도 /api/analysis 경제성 API

입력:
- 터널 길이
- 건설비
- 연간 편익
- 연간 운영비
- 운영연수
- 할인율

계산:
- 연간 편익과 운영비를 연도별 할인
- 현재가치 비용 = 건설비 + 운영비 현재가치
- B/C와 NPV 계산

[발표 주의]
- tunnel_length_km 입력은 스키마에 있지만 analyze_project 계산식에서는 사용하지 않는다.
- 이 API는 현재 주 후보 생성 화면에서 호출하지 않는다.

11.5 발표에서 반드시 조심해서 말해야 할 부분

1) “최적 노선”이 아니라 “MVP 예비 후보 노선”이라고 설명해야 한다.

2) “교통수요 예측”이라고 단정하기보다 “OD 가중 수요를 후보축 우선순위에 활용”한다고
   설명해야 한다.

3) estimated_flow는 실제 신규 노선 이용량이 아니다. 전환률을 적용한 diverted_flow도
   단순 규칙 기반 추정이다.

4) 자체 경제성 점수는 공식 예비타당성조사 지표가 아니다.

5) B/C에는 현재 유지관리비가 포함되지 않는다.

6) 토피고 판단 모델은 있으나 실제 토피고 값 생성은 현재 구현되지 않았다.

7) 암반등급 추정 코드는 있으나 현재 로컬 지질 테이블이 없어 실제 후보 계산에는
   반영되지 않는다.

8) 교량은 구현되지 않았다.

9) 지역 필터는 행정경계 polygon이 아니라 bounding box다.

10) 기존도로 오분류 문제가 확인되어 있다.
    - 기본 DEM 격자가 500m인데 구간 양 끝점만 각각 기존도로 50m 이내인지 검사한다.
    - 두 점이 서로 다른 도로에 가까워도 그 사이 전체를 기존도로로 분류할 수 있다.
    - centerline snap도 각 점을 독립적으로 가장 가까운 링크에 붙이며 연결성을 확인하지 않는다.
    - 실제 최근 결과 R001-D-S002는 4개 점이 서로 다른 비연속 도로 링크에 붙었지만
      1.921km 전체가 기존도로로 분류되었다.
    - 기존도로 판정에 동일·연결 링크 경로 검증을 추가해야 한다.

11) 최종 터널 총비용은 현재 구간별 암반계수 대신 기본 지반계수 1.30을 사용한다.

12) 유지관리비 포함 보고서 총사업비와 후보 B/C 비용 기준이 다르다.

13) 소스 일부 사용자 메시지와 문서에 인코딩이 깨져 보이는 문자열이 있으며,
    .env.example에는 해결되지 않은 merge conflict marker가 남아 있다. 배포 전 정리가 필요하다.

13. 발표자료용 요약
======================================================================

13.1 한 줄 서비스 소개

ONESTEP은 교통 OD 수요와 공간 데이터를 결합해 도로·터널 예비 후보 노선을 자동 생성하고,
초기 공사비·토지보상비·시간 및 거리 절감 편익을 한 화면에서 비교하는 사업 초기 검토
서비스이다.

13.2 핵심 문제

- 방대한 OD 수요에서 실제 검토할 연결축을 사람이 직접 찾기 어렵다.
- 후보 노선을 만들더라도 기존도로, 지형, 터널, 비용, 편익을 따로 검토해야 한다.
- 사업 초기에는 정밀설계 전에도 비교 가능한 후보군과 근거가 필요하다.

13.3 해결 방법

- OD 수요를 weighted K-Means로 압축해 수요 중심 후보 정점을 만든다.
- 정점쌍별 수요를 합산해 우선 검토 연결축을 선정한다.
- 기존 도로망 A*와 DEM 비용격자 A*를 함께 사용해 기준선과 신설 대안을 만든다.
- 구간을 기존도로·접속도로·신설도로·터널로 나눈다.
- 공사비, 필지별 토지보상비, 시간·거리 절감 편익, B/C와 자체 점수를 비교한다.
- 지도, 순위 카드, 템플릿 보고서로 결과를 제공한다.

13.4 기술 흐름

OD CSV·시나리오
→ 좌표 보완 및 지역 필터
→ 가중 K-Means 후보 정점
→ 정점쌍별 수요 집계
→ 기존 도로망 A* 기준선
→ DEM 비용격자 A* 대안
→ 도로·터널 구간 분류
→ 공사비·토지보상비 계산
→ 전환률·편익·B/C·점수
→ 지도·순위·예비 보고서

13.5 기대 효과

- 검토해야 할 전국 OD를 소수 후보축으로 압축
- 기존도로 활용안과 신설·터널 대안을 같은 기준에서 비교
- 수요, 지형, 비용, 편익을 하나의 초기 의사결정 흐름으로 통합
- 정밀조사 전에 추가 검토 가치가 있는 후보를 빠르게 선별
- 분석 과정과 경고를 결과에 남겨 후속 검토 항목을 명확히 제시

13.6 현재 한계

- 500m 기본 격자의 예비 선형
- 기존도로 연결성 검증 부족으로 오분류 가능
- 현재 지질 DB 미연결과 실제 토피고 미산정
- 교량 미구현
- 수요 전환률과 편익이 규칙 기반
- 유지관리비가 후보 B/C에 미반영
- 일부 환경 데이터 미연결
- 공식 타당성조사나 정밀설계를 대체할 수 없음
- PDF·PPT·생성형 AI 보고서 미구현

13.7 향후 개선 방향

- 도로 링크 연결성 기반 기존도로 판정
- 정밀 격자와 종단계획·토피고 계산
- 지질·하천·보호구역 데이터 완전 연결
- 구간별 암반계수 기반 터널비
- 교량과 구조물 비용
- 교통배정과 여객·화물 분리 편익
- 유지관리비를 포함한 일관된 생애주기 B/C
- 분석 저장, 후보 비교 차트, PDF/PPT 자동 생성
- 검증된 계산 결과를 기반으로 한 ChatGPT 발표자료 생성

- “ONESTEP을 정밀설계 시스템이 아니라 사업 초기 예비 후보 검토 MVP로 표현할 것”
- “공식 예비타당성조사 결과처럼 표현하지 말 것”
- “B/C에는 현재 유지관리비가 포함되지 않는다고 각주로 표시할 것”
- “지질·토피고·교량의 현재 상태를 과장하지 말 것”
- “기존도로 오분류 개선을 향후 핵심 과제로 포함할 것”
- “발표 슬라이드는 문제 → 데이터 → 후보 생성 → 노선 탐색 → 비용·편익 →
   결과 화면 → 한계 → 로드맵 순서로 구성할 것”

권장 슬라이드 구성 예시:

1. ONESTEP 한 줄 소개
2. 해결하려는 문제
3. 서비스 전체 사용자 흐름

5. OD 후보 정점과 연결축 생성
6. 기존도로 A*와 DEM A*
7. 도로·터널 구간 분류
8. 공사비와 토지보상비
9. 전환률과 편익
10. B/C와 후보 점수
11. 실제 화면과 사용자 조작
12. 현재 구현 상태
13. 현재 한계와 검증 필요사항
